# Color Report 1 — Current Data Audit

## Color Data Storage

### Where it lives
`dmc-data.js` — a plain JS file loaded via `<script>` tag on every page.

The data is stored as a top-level `const DMC_RAW` array of 5-tuples:
```js
const DMC_RAW = [
  ["blanc", "White",       255, 255, 255],
  ["ecru",  "Ecru",        240, 234, 218],
  ["310",   "Black",         0,   0,   0],
  // … 516 more
];
```

At the end of the file the array is mapped to structured objects and exported:
```js
const DMC = DMC_RAW.map(d => ({
  id:  d[0],
  name: d[1],
  rgb: [d[2], d[3], d[4]],
  lab: rgbToLab(d[2], d[3], d[4])   // computed at load time
}));
```

### Color format
- Stored values: **sRGB integers** (0–255) per channel.
- Derived values: **CIE L\*a\*b\*** computed via a standard D65 illuminant sRGB→XYZ→Lab conversion (`rgbToLab()` in the same file). LAB is memoised in a `Map` keyed by the 24-bit packed integer.
- No hex strings are stored; hex is constructed on-demand where needed (`#` + channel `toString(16)` padding).

### Total color count
**519 DMC entries** as of the last audit.  
DMC's commercial catalogue at time of writing contains approximately 500 standard Stranded Cotton (mouliné) colors, plus specialty items (metallic Light Effects, Color Variations, Satin, Coloris). A precise current list is not published by DMC but the standard count is widely cited as ~454–500 numbered colors plus `blanc`, `ecru`, and `B5200`. The app's 519 is plausible for a full standard catalogue with recent additions.

Notable inclusions:
- `blanc` (White), `ecru` (Ecru), `B5200` (Snow White) — three "white-family" special codes
- Numbered series 01–35 (new metallic-adjacent and special tones added ~2022–2024)
- Series 3880–3895 (2024 additions)
- Series 505–516, 730–734, 3880–3895 (added in commit `ba53bf1`, April 2026, by automated agent)

No Light Effects metallics (e.g., E310, E5200, E168) are present — these specialty threads with metallic shimmer are excluded, which is appropriate since their "color" is partially a surface-sheen effect not representable as a flat sRGB value.

### Data provenance — critical finding
**There is no attribution anywhere in `dmc-data.js`.** The file contains no comment indicating a source, methodology, author, date, or license.

Git history reveals:
| Commit | Date | Author | Change |
|--------|------|--------|--------|
| `5dd8b2b` | 2026-03-27 | google-labs-jules bot | First appearance — extracted from monolithic app during refactor |
| `40033d7` | 2026-03-28 | google-labs-jules bot | Unit tests for dE |
| `4d2c3a7` | 2026-03-29 | google-labs-jules bot | Added `dE2` squared-distance variant |
| `ba53bf1` | 2026-04-01 | google-labs-jules bot | Added B5200, 01–35, 3880–3895 |
| `4a3d038` | 2026-04-21 | Kate | Added 30+ more DMC colors and 188 Anchor colors |

The data was created by an automated agent extracting values from a pre-existing app. The ultimate origin of those RGB values is **unknown** — there is no link to a community dataset, spectrophotometer measurement file, or DMC publication. Values may have been hand-entered, copied from another dataset of unknown provenance, or generated by an AI model.

This is the single largest data quality risk: **we cannot validate the data against its source because the source is undocumented.**

## Anchor Color Data

Anchor colors live in `anchor-data.js`. This file is **significantly better documented**:

```
// Data sources used for reconciliation (retrieved April 2025):
//   [S1] Official Anchor colour card PDF / retailer scans
//   [S2] Stitchtastic DMC↔Anchor chart (stitchtastic.co.uk)
//   [S3] Cross-Stitched.com conversion table (cross-stitched.com)
//   [S4] Simone Balman's thread converter tool (sibalman/thread-converter on GitHub)
//   [S5] Joann/Michaels product photography (secondary reference, lower weight)
//
// Reconciliation methodology:
//   For each colour, RGB values from at least three of the sources above were
//   compared using CIEDE2000 (computed offline). Where all sources agreed within
//   ΔE < 3, the median was used and confidence is 'official'. Where sources
//   disagreed (ΔE 3–5 between any pair), the median was still used but
//   confidence is 'reconciled'.
```

Anchor data also documents **7 contested colors** where pairwise ΔE > 5 across sources, and emits `console.warn` for these at load time. The Anchor methodology is a reasonable multi-source reconciliation approach.

**Total Anchor entries: 606** (as of last update).

## How Colors Reach the Screen

### CSS swatches (palette panels, stash manager, tracker)
All swatch elements across every page use the same pattern:
```js
style={{ background: `rgb(${c.rgb[0]},${c.rgb[1]},${c.rgb[2]})` }}
```
This is a direct inline CSS `background` with **full opacity**. The `.colour-swatch` CSS class adds `border: 1px solid var(--border)` and `border-radius: 3px` — these do not alter color.

No `filter`, `opacity`, `mix-blend-mode`, or `backdrop-filter` is applied to swatch elements. The displayed swatch color is exactly the stored RGB value.

**Swatch rendering verdict: clean.** The pixel shown equals the stored value (within subpixel rendering noise of ±1 per channel).

### Canvas grid (Pattern Creator)
The main pattern grid is drawn on an HTML5 Canvas in `creator/canvasRenderer.js`.

Normal rendering (no overlay image):
```js
ctx2d.fillStyle = "rgba(" + fillRgb[0] + "," + fillRgb[1] + "," + fillRgb[2] + "," + alpha + ")";
ctx2d.fillRect(px, py, cSz, cSz);
```
Where `alpha = 1.0` when not dimmed, not highlighted, and no overlay is shown. **Canvas fills at full opacity = correct.**

Altered-alpha cases:
- **With overlay image** (`showOverlayImg = true`): `alpha = view === "both" ? 0.4 : 0.5`. At 0.4/0.5 opacity over a white background, DMC 310 (black) renders as approximately `rgb(153,153,153)` — massively different. This is intentional for the semi-transparent grid-over-image compositing feature, but it does mean the pattern grid is **not color-accurate when the reference overlay is shown**.
- **Highlight/dim mode**: non-highlighted cells are desaturated and dimmed. Intentional.

The canvas has no explicit `colorSpace` parameter — defaults to `'srgb'` in all modern browsers, which is correct.

**Canvas rendering verdict:** Correct at full opacity. Color-altered only during the overlay reference image workflow (intentional) and highlight/dim mode (intentional).

### No color space transformation
There is no ICC profile application, gamma correction beyond the standard sRGB linearisation (which is part of the LAB computation, not display rendering), or wide-gamut conversion anywhere in the pipeline.

## Summary of Known Issues

| Issue | Severity | Detail |
|-------|----------|--------|
| No data attribution | Critical | Origin of RGB values unknown — cannot validate against source |
| `blanc` = `B5200` = (255,255,255) | Critical | Two distinct threads rendered identically on screen |
| `02` = `318` = (171,171,171) | High | Two distinct threads rendered identically on screen |
| 4 other near-identical pairs | Medium | dE00 < 1.0 — visually indistinguishable |
| Color matching uses Euclidean LAB, not CIEDE2000 | Medium | Less perceptually accurate matching for blues/purples |
| Several notable RGB errors vs community reference | Variable | 666 is off by dE00=9.82; several others off by 3–5 |
| Canvas overlay mode alters color | Low | Intentional feature, but color-inaccurate during use |
