# B3 + B4 — Creator Sub-Page Consolidation + Materials Hub + Manager Shopping Tab

**Closes Quarter 2 (Proposal B).**

## Summary

- Pattern Creator collapses from five top-level sub-pages to three:
  `Pattern / Project / Materials & Output`. The legacy `prepare` /
  `legend` / `export` page IDs are auto-rewritten to `materials` with a
  sensible default `materialsTab`, so persisted prefs and existing
  bookmarks/links continue to work without migration code paths.
- A new `<CreatorMaterialsHub>` component renders inside the Materials &
  Output page and exposes four side-tabs: **Threads**, **Stash status**,
  **Shopping**, **Output**. The first three reuse the existing
  `LegendTab` / `PrepareTab` / `ExportTab` components unchanged (just a
  one-line guard rewrite); Shopping is a small new in-page aggregate.
- The Creator's right-rail `Sidebar` is now mode-aware within the page:
  hidden on Materials & Output (the hub takes the full content width),
  collapsed to a "Project at a glance" panel on Project, and unchanged
  on Pattern. Transition is a 160 ms opacity fade.
- The Stash Manager gains a new top-level **Shopping** tab between
  Pattern Library and the existing tools. It aggregates required threads
  across all *active* projects (`active` / `queued`), compares them
  against the stash, and exposes per-row "Add" + bulk "Add all deficits"
  CTAs that route through the existing `StashBridge.markManyToBuy` API.

## MaterialsHub sub-tab decisions

- Side-tab nav, top-of-panel rather than left-of-panel, to keep the
  reading width of the existing PrepareTab / LegendTab / ExportTab
  layouts unchanged. (A vertical rail would have forced a third-column
  reflow inside content that already does its own internal columns.)
- Active sub-tab persists via `UserPrefs.set('creator.materialsTab', ...)`.
- Default sub-tab on first visit is **Threads** (the most-visited of
  the three former tabs in our existing usage shape — see B5 dashboard
  card weighting). On second visit users land where they left off.
- The Shopping panel runs entirely on the in-memory pattern data already
  hydrated for the Threads/Stash views; no extra IndexedDB roundtrip.

## Manager Shopping aggregation rules

1. `ProjectStorage.listProjects()` lists meta records.
2. Each meta is checked against `ProjectStorage.getProjectStates()`;
   only `active` / `queued` (and unset, which defaults to active)
   projects are kept. `paused`, `complete`, `design`, `wishlist` are
   excluded.
3. Each retained project is lazy-loaded via `ProjectStorage.get(id)`
   (B5 PERF NOTE pattern — at most one full payload per active project).
4. For every palette entry, skein need is derived via `stitchesToSkeins`
   (or `skeinEst` fallback). Blends split into both component IDs.
5. Aggregated by composite key (`dmc:310` / `anchor:403`); summed across
   projects; compared against `StashBridge.getGlobalStash()`.
6. Rows with `deficit > 0` are returned, sorted by `deficit` descending.
7. Empty states distinguish "No active projects yet." from "All your
   active projects have the threads they need." (both British).

## Deferred items

- **ZIP export** (PDF + thread list + stash-add JSON bundle): out of
  scope for this epic. Rationale: there is no zip dependency in the
  repo today; pulling one in via CDN would re-introduce the offline-PWA
  fragility we deliberately avoid (no external runtime deps in the
  service worker), and writing a from-scratch DEFLATE/CRC implementation
  is a self-contained engineering task larger than the rest of B3+B4
  combined. Recommend: a follow-up ticket scoped specifically to
  `JSZip` (or equivalent) with explicit PWA-cache implications captured.
- **Per-row "Move to inventory" CTA on the Manager Shopping tab**: the
  brief asks for "add to shopping list" only; an "I bought it, move to
  owned" affordance is a natural follow-up but isn't in the brief and
  the Threads tab already supports the inverse flow.
