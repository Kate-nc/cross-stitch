# Import 2 — DMC Pattern File Structural Analysis

> Phase 1, Step 2. Deep structural analysis of every example pattern PDF
> the user supplied, plus two non-DMC PDFs from `TestUploads/` for
> cross-publisher contrast. All numerical claims come from the operator
> stream + text content dumps in [reports/import-2-raw/](import-2-raw/),
> generated by [scripts/analyse-dmc-pdfs.js](../scripts/analyse-dmc-pdfs.js)
> and [scripts/analyse-extra-pdfs.js](../scripts/analyse-extra-pdfs.js).

---

## TL;DR

- DMC ships **two distinct PDF layouts** in the corpus, both authored in **Adobe InDesign 19/20** and produced by **Adobe PDF Library 17.0**. Page size is always A4 portrait (595.3 × 841.9 pt).
- Both layouts use **filled-vector cells** for the chart — no symbols, no glyphs, no embedded raster bitmap of the grid. Each stitch is a coloured rectangle drawn with `setFillRGBColor` + `re` + `f`. This is **fundamentally different** from PatternKeeper / flosscross PDFs (which are single-character glyphs at grid coordinates) and from the existing app's experimental [pdf-importer.js](../pdf-importer.js).
- The chart is **not on page 1**. It is on either page 1+2 (split) or page 3 of 3, depending on layout. The current [pdf-importer.js](../pdf-importer.js) hard-codes "single-page glyph chart" assumptions and will produce nothing useful for either DMC layout.
- The legend is rendered as a small **vector table** with coloured swatch + DMC code + "x N skeins" rows. Stitch type ("cross stitch / point de croix", "backstitch / point arrière") and strand count ("Use 3 strands") are plain text labels above each sub-table.
- All publisher metadata is **bilingual or trilingual** (EN/FR ± ES). Any text-driven heuristic must accept translated synonyms.
- For comparison: **flosscross-exported PDFs** (Books and Blossoms) are a third pattern entirely — symbol-glyph charts with ~16 700 single-character text items per page and a clean tabular legend. The existing PK importer will already handle these.

## B. Layout A — split vector chart (PAT1968)

### B.1 Page roles

Per-page op counts come from [PAT1968_2.analysis.json](import-2-raw/PAT1968_2.analysis.json).

| Page | Role | text | longText | lineTo | rect | fillRGB | fill | stroke | images |
|---|---|---|---|---|---|---|---|---|---|
| 1 | **Chart — colour fills layer** | 34 | 16 | 24 513 | **6 334** | **5 880** | **8 072** | 5 356 | 0 |
| 2 | **Chart — backstitch + grid layer** | 33 | 15 | 24 513 | 983 | 7 | 2 727 | 2 714 | 0 |
| 3 | Getting started / tool kit | 61 | 39 | 179 | 5 | 5 | 22 | 2 | 0 |
| 4 | **Legend / colour key** | 88 | 57 | 393 | 111 | 52 | 103 | 114 | 0 |

The "1/4 … 4/4" page-count tokens are visible as `longText` items on every page (e.g. `"1/4"` at top of page 1).

### B.2 Page 1 — coloured cell layer

- 6 334 explicit `re` (rectangle) ops + 8 072 `fill` ops + 5 880 distinct `setFillRGBColor` calls.
  - The number of `fill` > `re` because the rendered grid lines/borders are also drawn as filled paths, not strokes.
  - The number of `setFillRGBColor` ≈ number of *colour changes*, not unique colours (the renderer probably re-states the colour per cell rather than batching).
- The 24 513 `lineTo` ops include cell-border construction (each `re` lowers to 4 lineTos in the operator stream) plus the chart's bounding frame and the "every 10 cells" major-grid emphasis.
- Zero `paintImageXObject` — **no embedded chart bitmap**. This is critical: a DMC chart cannot be extracted by saving the page as PNG and OCRing it; the PDF *is* already vector, we just need to read the operators.

### B.3 Page 2 — backstitch + grid layer

- Op profile is dramatically different: only **7** `setFillRGBColor` calls (versus 5 880 on page 1) and only **983** `re` ops, but **2 714** `stroke` ops.
- This is the **monochrome chart layer** — same cell geometry, same `lineTo` count (24 513, exactly equal to page 1, because both pages share the underlying grid template), but with colour fills suppressed and back-stitch / specialty lines drawn as strokes instead.
- Practical implication: a DMC parser only needs to scrape **one** of the two grid pages for cell colours (page 1) and the *other* (page 2) for backstitch geometry. Trying to scrape both blindly will double-count cells.
- This is the closest DMC gets to a "symbols-on-grid" view, and it is where any backstitch / specialty-stitch extraction must happen.

### B.4 Page 3 — instructional

- Only 22 fills, 2 strokes — entirely text + a few decorative icons.
- Long text dump: `"tool kit / fournitures"`, `"PEARL COTTON ART. 115ar/8 / COTON PERLÉ ART. 115ar/8"`, `"CROSS STITCH FABRIC / AIDA 14ct - GD1436BXA COL.336"`, `"DESIGN SIZE / DIMENSIONS DESSIN"`, `"18 cm / 7 ''"`, plus a multi-paragraph fabric-count explainer in EN and FR.
- Useful for metadata (fabric count, fabric SKU, finished size, recommended thread brand) but contains **no pattern data**.

### B.5 Page 4 — legend / colour key

Long text dump (cleaned and grouped):

```
header row (DMC coton perlé 115ar/8 - dmc perlé metallisé 315):
  symbol / symbole | colour / couleur | nº skeins* / d'échevettes*

cross stitch / point de croix:
  310, 930, 932, 5852, 5310, 5283, 5317     (with x1 next to each)

backstitch / point arrière (Use 3 strands - Utiliser 3 brins):
  5310

footnote:
  * Based on stitching with the number of strands shown above, onto 14 count Aida.
```

Observations:

- The `"x1"` markers appear **7 times** in the long-text stream — one per cross-stitch row in the legend, exactly matching the DMC code count (310, 930, 932, 5852, 5310, 5283, 5317).
- The `"symbol / symbole"` column header is real, but **the symbol cell itself is rendered as a coloured swatch via vector fill**, not as a glyph or text. There is nothing in the text stream to read for the symbol — we must read `setFillRGBColor` on a small rectangle in the legend region and trust positional alignment with the DMC code in the same row.
- Strand count is encoded in a label above the sub-table (`"Use 3 strands - Utiliser 3 brins"`), not per-row. There is one such label per stitch-type sub-table.
- Codes prefixed with a **leading digit `5`** (5310, 5283, 5317, 5852) are *not* regular DMC mouliné — they are perlé metallisé "Art. 315" SKUs. The art-line specifier `"DMC PERLÉ METALLISÉ 315"` appears as a sub-header above the rows that use those codes. **The parser must propagate this art-line context to each row** so the resulting palette entries carry the correct thread family, otherwise we will look up "5310" in the regular DMC mouliné table and either fail or return the wrong physical thread.

## C. Layout B — combined chart+legend (PAT2171)

### C.1 Page roles

Per [PAT2171_2.analysis.json](import-2-raw/PAT2171_2.analysis.json):

| Page | Role | text | longText | lineTo | rect | fillRGB | fill | stroke | images |
|---|---|---|---|---|---|---|---|---|---|
| 1 | **Title page (vertical text + raster preview)** | 113 | 17 | 182 | 9 | 10 | 24 | 7 | **2** |
| 2 | Getting started / tool kit | 123 | 54 | 179 | 7 | 7 | 22 | 2 | 0 |
| 3 | **Chart + legend combined** | 141 | 73 | **8 803** | **2 696** | **1 731** | **2 864** | 1 703 | 0 |

### C.2 Page 1 — title with embedded raster

- **Two `paintImageXObject` calls** — one for the DMC logo, one for a marketing photo of the finished design.
- The 40 `singleChars` here are not chart cells; they are individual letters of vertical text rotated by InDesign (look at the per-char `transform` — `scaleY` decreases monotonically from 7.186 down to negative values, indicating a near-90° rotation). Each character has its own text matrix because InDesign typesets curved/diagonal text per-glyph.
- **Heuristic warning:** any "single-char text item count > N → glyph chart" detector must filter out rotated text. Use the per-item `transform[1]` (skewY) and `transform[2]` (skewX) to reject anything that isn't an upright glyph at scale-Y ≈ font-size.

### C.3 Page 2 — instructional

- Same shape as Layout A page 3. Useful long text: `"dmc mouliné art. 117 embroidery thread"`, `"dmc mouliné light effect thread art 317w"`, `"dmc diamant"`, `"aida 5,5 pts/cm - aida 14 ct - GD1436BXI col.4015"`, `"design size / dimensions dessin"`, `"14 x 13 cm / 5.51 x 5.11 in"`.
- Trilingual label tokens (EN / FR / ES) are split across multiple `text` items because InDesign places each language as a separate run.

### C.4 Page 3 — chart + legend on one page

- 2 696 rect ops + 1 731 distinct fill colours + 2 864 fills → smaller chart than Layout A's page 1 (6 334 rects), consistent with this being a smaller design (14 × 13 cm finished vs 18 cm).
- The legend table sits **below** the chart on the same page. Long text shows the colour codes interleaved twice — once for cross-stitch (`B5200, 453, 3860, 09, 3799, E334, E3843, E825, D225`) and once for backstitch (`B5200, D225`).
- Special tokens to handle:
  - `"B5200"` — DMC mouliné special art. 117, "Snow White" (DMC's only B-prefixed code).
  - `"E334"`, `"E825"`, `"E3843"` — DMC Light Effects art 317w (E-prefix indicates the metallic / fluorescent line).
  - `"D225"` — DMC Diamant art 380 (D-prefix indicates Diamant). 
  - `"09"` — note **two-digit** code, which violates the "DMC IDs are 3-digit" assumption embedded in regex like `\b\d{3}\b`.
- Sub-table headers in the legend mirror Layout A: `"dmc mouliné spécial art.117"`, `"dmc diamant art.380"`, `"dmc mouliné spécial art.117"` (two appearances because the cross-stitch sub-table has art.117 codes followed by Light Effect art.317w codes, then a separate Diamant sub-table).
- Per-row strand count: `"Utilisez 1 brin / Use 1 strand"` and `"Utilisez 2 brins / Use 2 strands"` appear as labels alongside the columns. Same per-section pattern as Layout A but with three localisations.
- New legend column: `"points / stitches / puntos"` — a stitch *count*, not just `"x1"` skein markers. Layout B reports per-thread cell totals, which we can cross-validate against our extracted cells.

## D. Cross-publisher comparison: flosscross PDF (Books and Blossoms)

Per [Books_and_Blossoms.text.txt](import-2-raw/Books_and_Blossoms.text.txt):

| Page | Role | text | singleChars | fills | strokes | images |
|---|---|---|---|---|---|---|
| 1 | **Symbol chart** | 41 739 | **16 733** | 0 | 0 | 0 |
| 2 | Legend table | 158 | 30 | 0 | 0 | 0 |

Defining traits (versus DMC):

- Producer is `jsPDF 3.0.1`, creator is `flosscross - 118`. Everything is generated client-side from a digital pattern — no marketing pages, no instructional pages, no decorative vector art.
- The chart is **glyph-based**: every cell is a single character text item placed at a grid coordinate. The single-char dump shows characters like `"W"`, `"|"`, `"o"`, `"5"`, `"u"`, `"¨"` clustered at regular pixel intervals (e.g. `fs=7.2`, x-step = 8 pt) — exactly the layout the existing [pdf-importer.js](../pdf-importer.js) was written to handle.
- The legend on page 2 lists `"DMC 22 / Alizarin"`, `"DMC 167 / Yellow Beige - Very Dark"`, etc. — **English colour names included**, which the DMC publisher PDFs do not provide.
- A first column of single-char glyphs on the legend page (`"u", "5", "s", "m", "c", "b", "º", "|", "k", "p", "¨", "i", "d", "j", "*", "K", "q", "W", ",", "g", "@", ")", "¼"`) is the symbol → DMC mapping table itself, drawn as text. **This is the per-document glyph→DMC dictionary** that any glyph-chart parser needs to extract first.

## E. Cross-file consistency (DMC-specific conventions)

What we can rely on across DMC PDFs:

1. **Producer string contains `Adobe PDF Library`** and the creator is always `Adobe InDesign`. Useful as a publisher fingerprint.
2. **Page size is always A4 portrait** (595.3 × 841.9 pt). Other publishers use Letter or square aspect ratios.
3. **Page-count token** `"N/M"` appears in the long-text stream of every page (`"1/4"`, `"2/4"`, `"3/3"`, etc.).
4. **Footer is identical on every page** (`"dmc library - bibliothèque dmc"`, `"www.dmc.com © YEAR"`, hashtags, social channels). This footer can be used as a "this is a DMC publisher PDF" sniffer.
5. **Title appears as the same long-text run on every page** (`"moonlight | fleurs lunaires"`, `"plume de geai | jay feather"`). The first non-page-marker non-footer multilingual run on page 1 is the design title.
6. **Cell rendering is always vector fill, never glyph, never raster bitmap** — across both layouts in the corpus.
7. **Legend always lists DMC codes as plain text** with sub-headers naming the art line (`"dmc coton perlé 115ar/8"`, `"dmc mouliné spécial art.117"`, `"dmc diamant art.380"`, `"dmc perlé metallisé 315"`, `"dmc mouliné light effect thread art.317w"`).
8. **Stitch-type group labels are bilingual or trilingual** (`"cross stitch / point de croix / punto cruz"`, `"backstitch / point arrière / punto atrás"`).
9. **Strand-count label sits above each sub-table** (`"Use 3 strands - Utiliser 3 brins"`, `"Use 1 strand"`, `"Use 2 strands"`). One label per stitch-type per art-line.
10. **No DRM, no AcroForms, no XFA, no signatures, no encryption**. `IsAcroFormPresent: false`, `EncryptFilterName: null`, `IsXFAPresent: false`.

What varies (per-document detection required):

| Variable | Layout A (PAT1968) | Layout B (PAT2171) |
|---|---|---|
| Total pages | 4 | 3 |
| Chart page(s) | 1 (colours) + 2 (backstitch) | 3 only |
| Legend page | 4 | 3 (same as chart) |
| Title page | none separate | 1 (with raster preview) |
| Embedded raster images | 0 | 2 (logo + photo on page 1) |
| Languages | EN/FR | EN/FR/ES |
| Stitch counts in legend | no (just "x1" skein markers) | yes (`"points / stitches / puntos"` column) |
| Art lines used | Coton perlé 115ar/8, Perlé metallisé 315 | Mouliné spécial 117, Light Effect 317w, Diamant 380 |
| Code prefixes seen | digits, leading `5` for perlé metallisé | digits, `B`, `E`, `D`, **2-digit `09`** |

## F. Implications for the importer

These findings drive the architecture in [import-5-architecture.md](import-5-architecture.md). Key constraints any parser must satisfy:

1. **Chart-page detection cannot rely on page index** — must use op-density heuristics (rect count > 500 + setFillRGBColor count > 50 + image count = 0 ≈ vector chart).
2. **Two-layer chart support** — Layout A separates colour fills and backstitch into adjacent pages with identical geometry. The parser should detect this case and merge the layers, not import them as two separate patterns.
3. **Vector cell extraction**, not OCR — DMC PDFs need an operator-stream walker that pairs `setFillRGBColor` with the next filled `re`. No glyph recognition required.
4. **Grid inference from sparse cells** — DMC does not draw an empty/background cell; only stitched cells are rendered. The grid origin and cell size must be inferred by clustering the bottom-left `(x, y)` coordinates of every filled rect (modal x-step → cell width; modal y-step → cell height; min x → grid origin x).
5. **Legend extraction must be art-line aware** — propagate art-line headers (`"dmc mouliné spécial art.117"`, `"dmc perlé metallisé 315"`, etc.) down to the rows beneath them so codes resolve to the right thread family. Do not assume all codes are mouliné.
6. **Code regex must accept** `\d{2,5}` plus prefixes `B`, `E`, `D`, `ECRU`, `BLANC`, `B5200`, `BLANC NEIGE`. Strict `\b\d{3}\b` rejects valid codes (`09`, `B5200`, `D225`).
7. **Trilingual synonyms are non-optional** — match `cross stitch | point de croix | punto cruz` etc. as equivalence classes, both for stitch-type detection and for "this is a DMC PDF" sniffing.
8. **Symbol → colour mapping in the legend is positional, not textual** — for DMC PDFs, the swatch is a coloured rectangle drawn in the legend, not a character. Pair each row's swatch fill colour with the row's text content by `y`-coordinate proximity.
9. **Skein and stitch counts are validation gold** — Layout B publishes per-thread stitch counts. After extracting the chart, sum cells per palette entry and compare against the legend's count column. Any mismatch > 5 % is a parse confidence signal.
10. **The flosscross / PatternKeeper glyph case is real and separate** — the existing [pdf-importer.js](../pdf-importer.js) handles it (or at least targets it). The new DMC parser is an *additional* strategy under the same dispatch entry-point, not a replacement.
