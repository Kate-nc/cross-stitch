# C7 — Image Import Wizard (Plan)

> Source ticket: [reports/ux-8-post-B-audit.md](ux-8-post-B-audit.md#L277) — *"Image-import wizard rework"* (Quarter C, size **L**, descended from the original A2 still flagged red).

## 1. Executive Summary

- The ticket's own wording ("replace the single-step flow in `creator/PrepareTab.js`") is inaccurate. [creator/PrepareTab.js](../creator/PrepareTab.js#L1-L75) is the shopping-list / fabric-calculator tab; the actual single-step image-to-pattern UI lives in the file-loader at [tracker-app.js](../tracker-app.js#L2566-L2589) plus the parameter modal anchored on `importDialog === "image"` (state declared at [tracker-app.js](../tracker-app.js#L801-L802)). This plan addresses the real flow and renames the new module accordingly.
- The current path dumps every parameter — name, w, h, aspect-lock, fabric count, max colours, skip-bg, bg-tolerance — into one modal and hides dithering, blends, edge-enhancement, brightness/contrast/saturation entirely (see `runGenerationPipeline` call site in [creator/useCreatorState.js](../creator/useCreatorState.js#L815-L843)).
- Replace it with a five-step guided wizard (crop/orient → palette choice → size & fabric → preview/tune → confirm) rendered as a new component `creator/ImportWizard.js` plus per-step files, all bundled by [build-creator-bundle.js](../build-creator-bundle.js#L12-L48).
- Generation work continues to run on [generate-worker.js](../generate-worker.js#L1-L40); the wizard adds a *low-resolution* preview pass after step 4 using the existing pipeline so the user sees the result before committing.
- Wizard state persists to `localStorage` under `crossstitch_import_wizard_v1` so a refresh/return resumes mid-flow; abandoned wizards are evicted on a 7-day TTL to avoid quota bloat.

## 2. Goal

Replace the all-at-once parameter modal with a guided wizard that breaks the image-to-pattern decision into five small, recoverable steps. Each step shows a live preview thumbnail, sensible defaults derived from the source image, and a clear back/next path. The wizard reuses the existing pipeline (`runGenerationPipeline`, `generate-worker.js`) — no algorithmic changes — and reuses the C8 coachmark visual language for the optional first-time hints, but the wizard itself is a deterministic UI flow, not an onboarding overlay.

## 3. Current PrepareTab Flow (line-cited)

The brief says "single-step image-to-pattern conversion in `creator/PrepareTab.js`". That file is *not* the import flow. Verified by inspection:

- [creator/PrepareTab.js](../creator/PrepareTab.js#L1-L4) — file header: *"Prepare tab: shopping list + fabric calculator. Reads from CreatorContext."* No image import code.
- [creator/PrepareTab.js](../creator/PrepareTab.js#L31-L70) — only computes `rows` of palette entries with skein needs from `ctx.pat` / `ctx.pal`. The pattern is already generated by the time PrepareTab renders.

The real single-step flow is:

1. File picker / drag-drop in [tracker-app.js](../tracker-app.js#L2566-L2589). For `format === "image"` it sets `importDialog = "image"` and stashes the `Image` object in state.
2. Modal mounted later in `tracker-app.js` (the `importDialog === 'image'` branch around line 5380, per [memories/session/c7-image-import-research.md](../memories/session/c7-image-import-research.md) — verify before edit) with eight controls: name, max-w, max-h, lock-AR, fabric ct, max colours, skip-bg, bg-tolerance.
3. On Generate, [creator/useCreatorState.js](../creator/useCreatorState.js#L815-L843) passes the settings to `runGenerationPipeline` (synchronous fallback) or to [generate-worker.js](../generate-worker.js#L1-L25) (worker path).
4. The result `{ mapped, pal, cmap, confettiData }` is written to context; user lands on the Pattern tab in [creator/PatternTab.js](../creator/PatternTab.js).

Hidden-but-supported settings the modal never surfaces: `bri/con/sat`, `dith`, `allowBlends`, `smooth/smoothType`, `stitchCleanup`, `orphans` (all listed in the worker call at [creator/useCreatorState.js](../creator/useCreatorState.js#L840-L843) and the worker contract in [generate-worker.js](../generate-worker.js#L9-L13)).

## 4. Proposed Wizard Steps (overview)

| # | Title | What user does | Defaults | Validation | Back / Next |
|---|---|---|---|---|---|
| 1 | Crop & orient | Drag/scale crop box, rotate 90°, mirror H/V | Whole image, no rotation, no mirror | Crop must have ≥ 32×32 source pixels | Back disabled (step 1); Next always enabled |
| 2 | Palette | Choose DMC full / personal stash / limited count + slider | DMC full, max 30 colours, allow blends on | Stash option only enabled if owned threads ≥ 5 | Back to step 1; Next enabled if at least one option selected |
| 3 | Size & fabric | Width × height in stitches, lock AR, fabric count, over-two | Auto-fit to ≤ 200 stitches longest side; 14 ct; AR locked | 10 ≤ w,h ≤ 300; warn if > 200×200 (memory cost) | Back / Next always enabled |
| 4 | Preview & tune | View low-res preview, toggle dither / saliency, adjust contrast | Dither on, saliency off, contrast 0 | Preview must have generated successfully (`pal.length ≥ 1`); else stay disabled | Back to step 3 (re-uses cached preview if inputs unchanged); Next runs full-res generate |
| 5 | Confirm | Read-only summary card + name field | Name = source filename (sans ext) | Name 1–60 chars | Back to step 4; "Generate" commits and closes wizard |

Progress chip strip across the top (`Step N of 5`) doubles as a click-target for previously-completed steps (forward jumps blocked until step is reached at least once).

## 5. Step Contents in Detail

### Step 1 — Crop & orient

- **Surface:** Source image rendered in a fixed 320×320 (mobile: 90vw square) viewport with a draggable crop rectangle (CSS-only — no extra library).
- **Controls:** Rotate 90° CW (`Icons.rotate` — *add if missing, 24×24, 1.6 stroke, currentColor*), Flip H, Flip V, "Reset crop". Aspect-ratio guide chips: *Free*, *1:1*, *4:3*, *3:4*, *16:9*. Selecting a chip constrains the drag.
- **Output to wizard state:** `{ crop: { x, y, w, h }, rotate: 0|90|180|270, flipH: bool, flipV: bool }`. The image itself is *not* re-rasterised here; the crop+orient is applied by drawing into a temporary canvas at preview/commit time (matches the existing `cx.drawImage(img, 0, 0, sW, sH)` pattern in [creator/useCreatorState.js](../creator/useCreatorState.js#L805-L808)).
- **Defaults:** crop = full image, rotate = 0, flips = false.
- **Validation:** post-orient crop must yield ≥ 32 px on the shorter side. Display inline error message (no `alert()`); disable Next.

### Step 2 — Palette choice

- **Three radio cards:**
  1. *Full DMC* — quantise against entire DMC catalogue.
  2. *Personal stash* — use only DMC IDs with `owned > 0` from the stash bridge ([stash-bridge.js](../stash-bridge.js)). Card disabled with helper text when stash is empty or `< 5`.
  3. *Limited palette* — slider 5–80 colours; defaults to 30 (matches current `setImportMaxColours(30)` at [tracker-app.js](../tracker-app.js#L801)).
- **Toggle:** "Allow 2-thread blends" (default on; matches `effAllowBlends` in [creator/useCreatorState.js](../creator/useCreatorState.js#L842)). Note that blends are disabled automatically if user picks the Personal-stash option with `owned < 2 distinct ids`.
- **Validation:** picking *Limited palette* requires colour count between 5 and 80 (matches `quantize` k bounds in [colour-utils.js](../colour-utils.js)).

### Step 3 — Size & fabric

- **Inputs:** Width (stitches), Height (stitches), Lock aspect (checkbox), Fabric count (select from `FABRIC_COUNTS` in [constants.js](../constants.js)), Over-two checkbox (mirrors PrepareTab's `effectiveFabric = overTwo ? fabricCt/2 : fabricCt` at [creator/PrepareTab.js](../creator/PrepareTab.js#L26)).
- **Live readout (right column):**
  - Finished size in inches/cm using selected fabric count.
  - Estimated total stitches = `w * h` (excluding skipped bg if step 4 sets it).
  - Estimated skeins (rough): `Math.ceil(w*h / STITCHES_PER_SKEIN_ESTIMATE)` — same constant `STITCHES_PER_SKEIN_ESTIMATE = 800` already in [creator/PrepareTab.js](../creator/PrepareTab.js#L7).
  - Estimated time at the user's current `stitchSpeed` pref.
- **Defaults:** auto-scale source so longest side fits in 200 stitches (preserves AR). Fabric ct = 14. Over-two = off.
- **Validation:** 10 ≤ w,h ≤ 300. Warn (non-blocking) if `w*h > 40_000` per [reports/code-quality-04-edge-cases.md](code-quality-04-edge-cases.md#L13).
- **Mobile inputs:** `inputMode="numeric"`, `step="1"`, `min/max` set, per the *Mobile Form Input Conventions* repo memory.

### Step 4 — Preview & tune

- **Surface:** 256×256 canvas showing a *quantised low-res preview* generated by reusing `runGenerationPipeline` with `sW/sH` capped at 96 (~25× faster than full-res). Spinner overlays during generation.
- **Tune controls (collapsible "Advanced" section, closed by default):**
  - Dither (on/off) — maps to `dith` in worker settings.
  - Brightness, Contrast, Saturation — sliders −50..+50, default 0; map to `bri/con/sat`.
  - Skip near-white background — toggle + tolerance slider (3–50). Mirrors current `importSkipBg` at [tracker-app.js](../tracker-app.js#L802).
  - "Show saliency map" — overlay debug visual using the existing `generateSaliencyMap` from [colour-utils.js](../colour-utils.js); purely informational, never persisted.
- **Re-preview policy:** any control change debounces 250 ms then re-runs the cheap preview. Cancel any in-flight preview via the `reqId` pattern already used in [creator/useCreatorState.js](../creator/useCreatorState.js#L810).
- **Validation:** Next disabled until preview has produced ≥ 1 palette entry (i.e., not "no stitches"). If preview returns empty, surface inline guidance: "All pixels were treated as background — try lowering tolerance or unchecking *Skip background*." (Honest copy, no shrug emoji.)

### Step 5 — Confirm

- **Summary card** (read-only) listing the chosen settings: name, dimensions, fabric, palette mode + count, blends on/off, dither on/off, skip-bg state, estimated stitches/time/skeins.
- **Single editable field:** project name (default = source filename minus extension; `maxLength=60`).
- **Primary action:** *Generate full-resolution pattern*. This:
  1. Closes the wizard.
  2. Calls the existing path in [creator/useCreatorState.js](../creator/useCreatorState.js#L815-L843) with the wizard's settings.
  3. Saves project via `ProjectStorage.save(...)` (path already used at [tracker-app.js](../tracker-app.js#L2543) for JSON imports).
  4. Routes to the Pattern tab (existing behaviour).
- **Secondary action:** *Back to tune* (returns to step 4 keeping all state).

## 6. State Persistence

- Wizard state shape:
  ```
  { v: 1, ts: <epoch ms>, srcDataUrl: <string|undefined>, step: 1..5,
    crop: {...}, palette: {...}, size: {...}, tune: {...}, name: <string> }
  ```
- Stored under `localStorage["crossstitch_import_wizard_v1"]` after every step transition (debounced 500 ms during edits).
- On wizard mount: if `srcDataUrl` matches the freshly-loaded image's hash *and* `ts` is within 7 days, offer a *Resume previous import?* prompt with *Resume* / *Start fresh* (no auto-resume — explicit consent).
- Per-step preview thumbnails (64×64 PNG dataURLs) stored under the same key so the progress strip can render mini-thumbnails. Capped at ~16 KB total to stay under any reasonable localStorage budget.
- TTL eviction: on Creator boot, any `crosstitch_import_wizard_*` key older than 7 days is removed.
- Quota safety: wrap writes in `try/catch`; on `QuotaExceededError`, drop the `srcDataUrl` and thumbnails (keep settings only) and log to console — never throw to the UI.

## 8. Reuse of C8 Coachmark Primitive

Per [reports/c8-plan-first-stitch-coaching.md](c8-plan-first-stitch-coaching.md#L17-L43), C8 ships an overlay primitive used to anchor short instructional callouts to a target element. The wizard reuses *only* the visual treatment:

- Same dimmed backdrop colour and z-index strategy.
- Same callout card styling (border-radius, shadow, padding tokens, typography scale).
- Same `Icons.x` close affordance.

The wizard does **not** reuse the coachmark *behaviour*: coachmarks are one-shot, dismissable, and gated by `UserPrefs.onboarding.coached.*` flags. The wizard is a deterministic, modal flow that always runs when the user imports an image. The two systems must therefore not share state — only CSS variables / classnames. Concretely:

- Add a shared CSS class (e.g. `.cs-overlay-card`) in `styles.css` that both the coachmark primitive and the wizard's step shell consume, so future visual tweaks land in one place.
- The first-time-only "Welcome to image import" hint *inside* step 1 (anchored to the crop handles) **is** a coachmark, and uses the C8 primitive directly with flag `onboarding.coached.import_wizard`.

## 11. Mobile (≤ 480 px) Considerations

- Step shell is full-screen on mobile (no centred modal); progress strip becomes a horizontal dotted indicator with `Step N of 5` text label below.
- Crop viewport sized to `min(90vw, 90vh - 220px)` to leave room for controls + sticky footer.
- Sticky footer with *Back* (left) and *Next/Generate* (right), 48 px min height for tap targets.
- All number inputs declare `inputMode="numeric"` and `step/min/max`; sliders have `aria-valuemin/max/now` (per *Mobile Form Input Conventions* memory).
- Step 4 preview canvas uses `image-rendering: pixelated` and capped at `min(256, 90vw)`.
- The "Resume previous import?" prompt is a regular modal, not a coachmark, so it survives orientation change.

## 12. Accessibility

- Wizard root is `role="dialog"` `aria-modal="true"` `aria-labelledby` pointing at the current step's `<h2>` title. Updated on each step change.
- Initial focus on step mount: the first interactive control. Focus is *moved* (not just visible) on every step transition — both Forward and Back.
- Progress strip uses `role="list"` with each step a `role="listitem"`; the active step has `aria-current="step"`. Completed steps are buttons (focusable, can jump back); unreached steps are non-interactive `<span>` with `aria-disabled="true"`.
- Escape key: prompts "Discard import? Your settings are saved for 7 days." Two buttons; Cancel returns focus to the previously-focused control.
- Live region (`aria-live="polite"`) announces "Step 3 of 5: Size and fabric" on each transition.
- Sliders have visible value labels (not just thumbs); colour-only state never the sole indicator.
- Error messages are `role="alert"` and announced.
- Contrast: wizard chrome inherits the app's accent token (`--accent`) which apply-prefs.js already validates against WCAG AA per [apply-prefs.js](../apply-prefs.js#L1-L80).

## 14. Risks

- **Worker thread coupling.** Step 4's preview re-uses `generate-worker.js`. Spamming previews while the user drags a slider could swamp the worker queue. Mitigation: 250 ms debounce + `reqId` cancellation (existing pattern at [creator/useCreatorState.js](../creator/useCreatorState.js#L810-L811)), and the worker drops messages whose `reqId` is stale.
- **Abandoned wizards filling localStorage.** A 5 MB image dataURL persisted across many sessions would blow the quota. Mitigation: 7-day TTL, drop `srcDataUrl` first on `QuotaExceededError`, and store only a hash of the source for the resume check rather than the full image when possible.
- **Pipeline drift.** `runGenerationPipeline` is shared between full and preview runs ([creator/generate.js](../creator/generate.js#L19) header note). If a future PR changes the preview path independently, the wizard's preview will diverge from the committed result. Mitigation: regression test that asserts preview palette ⊆ full-run palette for a fixture image.
- **Mobile crop on iOS Safari.** Touch-event coordinate math differs from mouse on Safari < 16. Mitigation: use Pointer Events with `touch-action: none` on the crop rectangle; fall back to a numeric crop input panel if `PointerEvent` is undefined.
- **Two import entry points still exist** (Home screen "From image" + Creator). Both must mount the same wizard component or users will see inconsistent UX. PR-1 audits both call sites.
- **Help drawer / coachmark coupling.** If C8 ships first and changes the overlay class names, this wizard's PR-1 baseline assumes the old names. Mitigation: PR-1 introduces the `.cs-overlay-card` shared class.

## 15. Open Questions

1. Should the wizard be available from the Tracker page (which currently owns the file picker via [tracker-app.js](../tracker-app.js#L2566-L2589)), or only from the Creator? The brief implies Creator only; current single-step modal lives in Tracker.
2. Should *Personal stash* mode disable blends entirely when stash has fewer than 2 distinct DMC ids, or attempt single-thread blends? Current pipeline supports the former cleanly; the latter requires a worker change.
3. Does the wizard need a "Save preset" feature (save settings as a named template for next time)? Out of scope for this plan but worth flagging.
4. Should the auto-fit longest-side default be 200 stitches or pull from `UserPrefs.creator.defaultMaxStitches` if such a pref exists? Need to check user-prefs.js.
5. Is `CREATOR_CACHE_KEY` bumped per-PR or only on the final cut-over? Repo memory implies per-PR is safer when bundle changes ship behind a flag.
6. Step 1 rotate/flip — should this be applied destructively to the source before generation, or carried as transform metadata? Plan currently picks the latter (re-applied at generate time) to avoid quality loss on round-trips.
